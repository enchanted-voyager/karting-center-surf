# Требования к MVP: клиентское приложение «Картинг-центр»

## 1. Контекст и границы поставки

Заказчик — Женя, владелец картинг-центра «Апекс». Сейчас запись на заезды
ведётся вручную через Telegram и доску, из-за чего возникают двойные брони
на один карт и простаивающие карты из-за поздних неявок/отмен.

**Что уже существует в инфраструктуре заказчика** (не разрабатываем):
бэкенд, хранение данных, формирование расписания, интерфейс
маршала-инструктора и интерфейс администратора/владельца.

**Что входит в текущую поставку:**

1. Клиентское мобильное приложение.
2. API для этого приложения.

Слоты, конфигурации трассы и маршалы приходят из существующего бэкенда
через API — приложение их только читает и не создаёт/не редактирует.
Гарантию «0 двойных броней» обеспечивает бэкенд (атомарная проверка
свободных карт при бронировании); приложение полагается на ответ API и
корректно обрабатывает отказ при отсутствии мест.

## 2. User Stories

1. **Просмотр расписания заездов**
   Я, как клиент, хочу видеть расписание заездов на ближайшие дни с
   указанием времени старта, конфигурации трассы (короткая/длинная) и
   количества свободных карт, чтобы выбрать удобный заезд.

2. **Бронирование заезда**
   Я, как клиент, хочу самостоятельно записаться на выбранный заезд и
   указать, использую ли свою экипировку (шлем, подшлемник) или беру её
   напрокат, чтобы не звонить администратору и не зависеть от переписки в
   мессенджере.

3. **Отмена бронирования**
   Я, как клиент, хочу иметь возможность отменить свою запись заранее,
   чтобы освободить место для других, при этом понимая, что отменить
   бронь прямо перед стартом уже нельзя.

4. **Оценка маршала после заезда**
   Я, как клиент, хочу после завершения заезда поставить оценку
   маршалу-инструктору, чтобы отметить качество проведения заезда.

## 3. Основные сценарии использования

### Сценарий 1: Просмотр расписания и выбор заезда

1. Пользователь открывает приложение и попадает в раздел «Расписание».
2. По умолчанию приложение показывает доступные заезды на ближайшие
   **7 дней** (горизонт планирования расписания у заказчика — «на
   неделю вперёд»); более поздние даты доступны через фильтр по дате.
3. Если на ближайшие дни расписания нет, отображается понятное
   пустое состояние («Пока нет доступных заездов»).
4. Для каждого заезда показываются: дата и время старта, конфигурация
   трассы (короткая для новичков / длинная для опытных), имя маршала
   (только для отображения) и количество свободных карт.
5. Пользователь выбирает интересующий заезд и открывает карточку с
   подробностями (длительность заезда 15–20 минут + инструктаж перед
   стартом, ограничение по числу участников на новичковых заездах — не
   более 8 человек).

### Сценарий 2: Бронирование заезда

1. Пользователь находится на карточке выбранного заезда и нажимает
   «Записаться».
2. Приложение предлагает указать экипировку: своя (шлем, подшлемник) или
   прокат — при выборе проката учитывается доступный прокатный фонд,
   получаемый из API.
3. Пользователь подтверждает запись.
4. Приложение отправляет запрос на бронирование в API:
   - при успехе — сохраняет бронь, показывает подтверждение и добавляет
     запись в «Мои записи»;
   - при отказе бэкенда (мест не осталось — кто-то забронировал раньше)
     — показывает понятное сообщение и предлагает выбрать другой заезд.
5. Если пользователь позже отменяет запись:
   - отмена заранее (до заезда) — бронь отменяется, слот освобождается;
   - отмена в последние 10 минут перед стартом — приложение блокирует
     отмену и поясняет, почему это невозможно.
6. Если заезд отменяет сам центр (организатор или из-за погоды), бронь
   клиента не удаляется, а получает статус **«Отменён центром»** с
   указанием причины; клиенту приходит push-уведомление. Повторная
   запись на отменённый слот запрещена. (Само действие отмены заезда
   выполняется в существующей инфраструктуре центра — вне скоупа
   клиентского приложения.)

## 4. Ключевые сущности данных (по контракту API)

- **Заезд** — дата/время старта, конфигурация трассы, маршал (read-only),
  число свободных карт, статус (открыт/отменён центром/завершён).
- **Конфигурация трассы** — короткая (для новичков) / длинная (для
  опытных).
- **Бронирование** — клиент, заезд, тип экипировки (своя/прокат),
  статус (активно/отменено клиентом/отменено центром/завершено).
- **Экипировка** — своя или из проката (с учётом доступного фонда).
- **Оценка маршала** — заезд, маршал, оценка (звёзды).

## 5. Вне рамок текущей поставки

- Интерфейсы маршала и администратора/владельца — уже существуют в
  инфраструктуре заказчика.
- Создание/редактирование расписания, конфигураций трассы и данных о
  картах — это ответственность существующего бэкенда.
- Онлайн-оплата — заказчик пока принимает наличные/переводы, вопрос
  открыт.
- Автоматический учёт погоды — решение об отмене заезда принимается в
  существующей инфраструктуре; приложение лишь отображает результат.
- Программа лояльности для постоянных клиентов.
- Push-напоминания о предстоящей записи (кроме уведомления об отмене
  заезда центром, см. Сценарий 2, шаг 6).
- Детали серверной интеграции (транзакционность, SLA, маппинг сущностей
  на внутренние модели бэкенда), миграция исторических данных.
