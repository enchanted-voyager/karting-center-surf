# Архитектурный план MVP: клиентское приложение «Картинг-центр»

## 1. Границы поставки

Разрабатываем только:

1. **Клиентское мобильное приложение** (iOS/Android).
2. **API** для этого приложения.

Уже существует у заказчика и не разрабатывается нами: бэкенд, хранение данных, формирование расписания, интерфейс маршала-инструктора, интерфейс администратора/владельца (админка). Существующий бэкенд выступает **источником истины** по слотам, конфигурациям трассы, маршалам и гарантирует «0 двойных броней» на своей стороне.

Наше API — это **BFF-слой (Backend for Frontend)**: он предоставляет мобильному приложению удобный контракт, а сам обращается к существующему бэкенду заказчика. Детали этой серверной интеграции (протокол, транзакционность, маппинг на внутренние модели бэкенда) — вне скоупа текущей задачи; для проектирования считаем существующий бэкенд «чёрным ящиком», отвечающим на запросы нашего API.

## 2. Общая схема компонентов

```
┌────────────────────┐        ┌──────────────────────┐        ┌───────────────────────────┐
│  Мобильное          │  HTTPS │   API (наша разработка)│        │  Существующий бэкенд        │
│  приложение          │ ──────▶│                        │ ──────▶│  заказчика (чёрный ящик)     │
│  (клиент)            │◀────── │  - Auth                │◀────── │  - хранение расписания        │
│                      │        │  - Schedule            │        │  - хранение слотов/трасс      │
│                      │        │  - Booking             │        │  - хранение маршалов          │
│                      │        │  - Rating              │        │  - проверка свободных карт     │
│                      │        │  - Notifications        │        │  - интерфейсы маршала/админки  │
└────────────────────┘        └──────────────────────┘        └───────────────────────────┘
                                        ▲
                                        │ push (FCM/APNs)
                                        ▼
                               ┌──────────────────┐
                               │  Push-уведомления  │
                               └──────────────────┘
```

## 3. Компоненты API

| Модуль | Ответственность |
|---|---|
| **Auth** | Регистрация/вход клиента, выдача токена сессии (JWT). Это новая сущность — учётные записи клиентов мобильного приложения, которых не было в прежнем ручном процессе (Telegram/доска). |
| **Schedule** | Чтение расписания заездов на ближайшие 7 дней (по умолчанию) + фильтр по дате. Проксирует/кеширует данные из бэкенда: время старта, конфигурация трассы, имя маршала (только для отображения), число свободных карт. |
| **Booking** | Создание и отмена брони. Пробрасывает запрос в бэкенд, который атомарно проверяет наличие свободных карт (гарантия «0 двойных броней» — на его стороне). Обрабатывает статус «Отменён центром». |
| **Rating** | Приём оценки маршала клиентом после завершённого заезда. |
| **Notifications** | Регистрация push-токена устройства, отправка напоминаний о записи и уведомлений об отмене заезда центром/по погоде. |

## 4. Основные потоки (по сценариям из требований)

**Сценарий 1 — просмотр расписания:**
приложение → `GET /schedule?from&to` → API запрашивает у бэкенда слоты на диапазон → возвращает список заездов (дата/время, конфигурация трассы, маршал, свободные карты) → если пусто, приложение показывает empty state.

**Сценарий 2 — бронирование заезда:**
приложение → `POST /bookings` (slot_id, тип экипировки) → API вызывает бэкенд → при успехе бэкенд резервирует карт и возвращает подтверждение → API сохраняет запись и возвращает её приложению → при отказе (мест не осталось) API возвращает понятную ошибку, приложение предлагает выбрать другой заезд.

**Отмена брони:**
приложение → `DELETE /bookings/{id}` → API проверяет через бэкенд, не позже ли это 10 минут до старта → бэкенд отменяет либо блокирует отмену → приложение показывает результат.

**Отмена центром/по погоде:**
бэкенд меняет статус слота → API получает обновление (вебхук/поллинг — деталь интеграции вне скоупа) → рассылает push всем клиентам с активной бронью на этот слот → в приложении бронь получает статус «Отменён центром» с причиной.

## 5. Верхнеуровневый список эндпоинтов API

```
POST   /auth/register
POST   /auth/login

GET    /schedule?from=&to=&track_config=
GET    /slots/{slotId}

POST   /bookings                 { slotId, equipment: "own" | "rental" }
GET    /me/bookings
DELETE /bookings/{bookingId}

POST   /bookings/{bookingId}/rating   { stars }

POST   /devices                  { pushToken, platform }
```

## 6. Простая схема данных (таблицы)

Часть сущностей — это данные, которыми **владеет существующий бэкенд** (наше API их только читает/проксирует, помечено «read-only»). Часть — новые сущности, которые появляются **только с мобильным приложением** и которыми управляет наше API (помечено «owned by API»).

### `clients` — *owned by API*
Учётные записи пользователей мобильного приложения (в старом процессе их не было).

| Поле | Тип | Описание |
|---|---|---|
| id | uuid PK | |
| phone | varchar | логин, уникальный |
| name | varchar | имя клиента |
| password_hash | varchar | |
| created_at | timestamp | |

### `devices` — *owned by API*
Push-токены для напоминаний и уведомлений об отмене.

| Поле | Тип | Описание |
|---|---|---|
| id | uuid PK | |
| client_id | FK → clients.id | |
| push_token | varchar | |
| platform | enum(ios, android) | |
| created_at | timestamp | |

### `slots` (заезды) — *read-only, зеркало бэкенда*
Кешируется в API для быстрого ответа приложению, источник истины — бэкенд.

| Поле | Тип | Описание |
|---|---|---|
| id | uuid PK | id слота в бэкенде |

| status | enum(open, cancelled_by_center, completed) | |

### `track_configs` — *read-only, зеркало бэкенда*

| Поле | Тип | Описание |

Наше API хранит бронь для показа в «Мои записи»; фактическую проверку свободных мест и атомарность гарантирует бэкенд при создании записи.

| Поле | Тип | Описание |

Прокатный фонд экипировки на заезд, для проверки при выборе «прокат».

| Поле | Тип | Описание |

## 7. Вне скоупа текущей поставки

- Реализация самого бэкенда, формирование расписания, хранение слотов/трасс/маршалов «по-настоящему» — уже есть у заказчика.
- Интерфейсы маршала и администратора/владельца — уже есть.
- Онлайн-оплата — вопрос открыт у заказчика, не проектируется в этом MVP.
- Автоматический учёт погоды и решение об отмене заезда — принимается в существующей инфраструктуре, приложение только отображает результат.
- Программа лояльности для постоянных клиентов.
- Миграция и дозаполнение старых данных.
